# Roadmap Template Revision

## Evidence Source

The revision is based on GDS-ROADMAP-001 Platform Evolution Roadmap Update.

GDS-ROADMAP-001 showed that roadmap authors need to record more than a topic
and future direction. They must also identify ownership, canonical repository,
promotion stage, evidence, architecture impact, dependencies, and validation.

## Revision Summary

Updated `templates/roadmap_template.md` to include:

- Template Metadata.
- Identity.
- Owner.
- Canonical Repository.
- Roadmap Type.
- Promotion Stage.
- Promotion Evidence.
- Promotion Criteria.
- Success Criteria.
- Architecture Impact.
- Dependencies.
- Validation Strategy.
- Guardrails.
- Completion Report Linkage.

## Lessons From ROADMAP-001

- Platform direction can be confused with implementation approval unless
  guardrails are explicit.
- GDS-owned roadmap direction must name the canonical repository and separate
  field project implementation ownership.
- Future candidates need promotion stage and promotion evidence.
- Center Pattern roadmap work needs architecture impact and boundary fields.
- AI Repository Index, README routes, roadmap index routes, and mojibake marker
  search should be part of validation strategy.

## Compatibility

The revised template preserves the existing Product Documentation Hierarchy
layers and previous roadmap fields while adding stronger governance fields.

Existing roadmap documents do not need to be rewritten by this Q.
